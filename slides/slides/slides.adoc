== The Cold Start Problem

[.en]
Knative scales to zero to save resources. When a request arrives, a **cold start** happens:

[.pt]
O Knative escala para zero para economizar recursos. Quando uma requisição chega, acontece um **cold start**:

++++
<div class="en">
<ol>
<li>Knative Activator receives the request</li>
<li>Pod is scheduled onto a node</li>
<li>Container image is pulled</li>
<li>Application starts</li>
<li>First request is served</li>
</ol>
</div>
<div class="pt">
<ol>
<li>O Activator do Knative recebe a requisição</li>
<li>O Pod é agendado em um nó</li>
<li>A imagem do container é baixada</li>
<li>A aplicação inicia</li>
<li>A primeira requisição é atendida</li>
</ol>
</div>
++++

[NOTE.speaker]
--
[.en]#With JVM apps, step 4 alone can take 2-3 seconds. That's before the request is even processed.#
[.pt]#Com apps JVM, só o passo 4 pode levar 2-3 segundos. Isso antes da requisição ser processada.#
--

== The Demo App

[.en]
Event Processor API with PostgreSQL:

[.pt]
API de Processamento de Eventos com PostgreSQL:

[source,java]
----
@Path("/api/webhooks")
public class WebhookResource {

    @POST
    @Transactional
    public Response create(@Valid WebhookRequest request) {
        Webhook webhook = new Webhook();
        webhook.source = request.source();
        webhook.eventType = request.eventType();
        webhook.payload = request.payload();
        webhook.receivedAt = Instant.now();
        webhook.persist();
        return Response.status(201)
            .entity(WebhookResponse.from(webhook)).build();
    }
}
----

== Why No Virtual Threads?

[.en]
Knative + virtual threads = **misaligned model**:

[.pt]
Knative + virtual threads = **modelo desalinhado**:

++++
<div class="en">
<ul>
<li><strong>Horizontal scaling</strong> handles concurrency, not vertical</li>
<li><strong>DB pool</strong> (~20 connections) is the real bottleneck per pod</li>
<li><strong>Jackson ThreadLocal</strong> adds per-vthread memory overhead</li>
<li><strong>Simplicity</strong> — blocking model is easier to reason about</li>
</ul>
</div>
<div class="pt">
<ul>
<li><strong>Scaling horizontal</strong> cuida da concorrência, não vertical</li>
<li><strong>Pool de DB</strong> (~20 conexões) é o gargalo real por pod</li>
<li><strong>Jackson ThreadLocal</strong> adiciona overhead de memória por vthread</li>
<li><strong>Simplicidade</strong> — modelo blocking é mais fácil de raciocinar</li>
</ul>
</div>
++++

[NOTE.speaker]
--
[.en]#Virtual threads shine when you have many concurrent blocking calls on a single JVM. In Knative, you scale out instead of up.#
[.pt]#Virtual threads brilham quando você tem muitas chamadas blocking concorrentes em uma única JVM. No Knative, você escala out ao invés de up.#
--
