== The Entity

`Webhook` extends PanacheEntity with snake_case naming strategy:

[source,java]
----
@Entity
public class Webhook extends PanacheEntity {

@Column(nullable = false)
public String source;

@Column(nullable = false)
public String eventType;

@Column(columnDefinition = "TEXT")
public String payload;

public Instant receivedAt;
public String processingMode;
}
----

== The API

[.en]
Input DTO with Bean Validation, output DTO with factory method:

[.pt]
DTO de entrada com Bean Validation, DTO de saída com factory method:

[source,java]
----
public record WebhookRequest(
@NotBlank @Size(max = 255) String source,
@NotBlank @Size(max = 100) String eventType,
String payload
) {}

public record WebhookResponse(
Long id, String source, String eventType,
String payload, Instant receivedAt, String processingMode
) {
public static WebhookResponse from(Webhook w) {
return new WebhookResponse(w.id, w.source,
w.eventType, w.payload, w.receivedAt, w.processingMode);
}
}

== The Cold Start Problem

[.en]
Knative scales to zero to save resources. When a request arrives, a **cold start** happens:

[.pt]
O Knative escala para zero para economizar recursos. Quando uma requisição chega, acontece um **cold start**:

++++
<div class="en">
<ol>
<li>Knative Activator receives the request</li>
<li>Pod is scheduled onto a node</li>
<li>Container image is pulled</li>
<li>Container is created and app starts</li>
<li>First request is served</li>
</ol>
<p>With JVM apps, <strong>step 4 alone takes 2-3 seconds</strong> before the request is even processed.</p>
</div>
<div class="pt">
<ol>
<li>O Activator do Knative recebe a requisição</li>
<li>O Pod é agendado em um nó</li>
<li>A imagem do container é baixada</li>
<li>O container é criado e a app inicia</li>
<li>A primeira requisição é atendida</li>
</ol>
<p>Com apps JVM, <strong>só o passo 4 leva 2-3 segundos</strong> antes da requisição ser processada.</p>
</div>
++++

== Why Cold Start Matters

[.en]
Our benchmark measures **container-based** startup, the same path Knative follows:

[.pt]
Nosso benchmark mede startup **baseado em container**, o mesmo caminho que o Knative segue:

++++
<div class="en">
<ul>
<li><strong>Container start:</strong> <code>docker run</code> → health check. The <em>production-realistic</em> metric: container creation + app init + health probe. What Knative users experience.</li>
<li><strong>Quarkus log:</strong> "started in Xs" from container logs. App-only, eliminates Docker overhead.</li>
<li><strong>Cold vs Warm:</strong> 1st run drops OS page cache. Warm runs reuse cached pages. <em>Scale-from-zero = cold.</em></li>
</ul>
<p>CPU pinning: app=cpuset 2-5 (4 vCPUs), DB=cpuset 0-1, k6=cpuset 6-11</p>
</div>
<div class="pt">
<ul>
<li><strong>Start do container:</strong> <code>docker run</code> → health check. Métrica <em>realista para produção</em>: criação do container + init da app + health probe.</li>
<li><strong>Log do Quarkus:</strong> "started in Xs" dos logs do container. Só a app, elimina overhead do Docker.</li>
<li><strong>Cold vs Warm:</strong> 1a exec limpa page cache do SO. Execuções quentes reusam cache. <em>Scale-from-zero = cold.</em></li>
</ul>
<p>CPU pinning: app=cpuset 2-5 (4 vCPUs), BD=cpuset 0-1, k6=cpuset 6-11</p>

== Quarkus Native to the Rescue

[.en]
Quarkus compiles to a **standalone native executable** via Mandrel/GraalVM:

[.pt]
O Quarkus compila para um **executável nativo standalone** via Mandrel/GraalVM:

++++
<div class="en">
<ul>

<li><strong>No classpath scanning</strong> at startup</li>
<li><strong>No JIT warmup:</strong> AOT compiled</li>
<li><strong>Sub-second startup,</strong> ideal for serverless</li>
<li><strong>Cold start ≈ warm start:</strong> no OS cache dependency</li>
<li><strong>20x less memory:</strong> 12 MB RSS vs 241 MB</li>
</ul>
</div>
<div class="pt">

== Production Impact: Cost & SLA

++++
<div class="en">
<ul>
<li><strong>SLA:</strong> 3,541ms JVM cold start breaches a 3s SLO on every scale-from-zero event. Native's 642ms fits inside most "fast" SLO budgets.</li>
<li><strong>Cost:</strong> 241 MB RSS per JVM pod = ~66 pods on a 16 GB node. 12 MB per Native pod = 1,300+ pods. For 500 replicas: JVM needs ~8 nodes, Native needs fewer than 1.</li>
<li><strong>Autoscaling bursts:</strong> JVM queues traffic for 3+ seconds per pod during scale-out. Native handles the same burst in under 600ms per pod.</li>
<li><strong>Beyond serverless:</strong> The cold start gap also hits rolling deployments, OOM kill restarts, and cluster autoscaler events.</li>
</ul>
<p class="note">Native is not just faster. It is cheaper to run and easier to keep within SLO.</p>
</div>
<div class="pt">
<ul>
<li><strong>SLA:</strong> Cold start de 3.541ms no JVM viola SLO de 3s a cada evento scale-from-zero. Os 642ms do Native cabem na maioria dos orçamentos de SLO "rápidos".</li>
<li><strong>Custo:</strong> 241 MB RSS por pod JVM = ~66 pods em 16 GB de nó. 12 MB por pod Native = 1.300+ pods. Para 500 réplicas: JVM precisa de ~8 nós, Native de menos de 1.</li>
<li><strong>Bursts de autoscaling:</strong> JVM mantém tráfego em fila por 3+ segundos por pod durante scale-out. Native absorve o mesmo burst em menos de 600ms por pod.</li>
<li><strong>Além do serverless:</strong> O gap de cold start também afeta rolling deployments, reinícios por OOM kill e eventos de cluster autoscaler.</li>
</ul>
<p class="note">Native não é só mais rápido. É mais barato para rodar e mais fácil de manter dentro do SLO.</p>
</div>
++++

== Throughput: The Trade-Off

++++
<div class="en">
<ul>
<li><strong>JVM wins throughput</strong> in our DB-bound workload (408 vs 260 req/s). JIT optimizes hot paths at runtime.</li>
<li>Native uses Serial GC with smaller heap, leading to <strong>higher tail latency under load</strong> (P99: 7,118 ms vs 4,265 ms)</li>
<li>Expected trade-off: <strong>Native trades peak throughput for startup speed and memory density</strong></li>
<li>On Knative, you run many small instances, <strong>not one at max throughput</strong></li>
</ul>
<p class="note">Both modes hit PostgreSQL on every request. The DB is the bottleneck. JVM's JIT advantage appears in sustained throughput; Native's Serial GC shows under concurrent load.</p>
</div>
<div class="pt">
<ul>
<li><strong>JVM ganha em throughput</strong> na workload limitada pelo BD (408 vs 260 req/s). JIT otimiza hot paths em runtime.</li>
<li>Native usa Serial GC com heap menor, gerando <strong>latência de cauda maior sob carga</strong> (P99: 7.118 ms vs 4.265 ms)</li>
<li>Trade-off esperado: <strong>Native troca throughput máximo por velocidade de startup e densidade de memória</strong></li>
<li>No Knative, você roda muitas instâncias pequenas, <strong>não uma no throughput máximo</strong></li>
</ul>
<p class="note">Ambos os modos acessam PostgreSQL a cada requisição. O BD é o gargalo. A vantagem do JIT aparece no throughput sustentado; o Serial GC do Native aparece sob carga concorrente.</p>
</div>
